# 网页端支持方案分析

## 一、概述

本项目使用 Flutter 开发，本身具备编译到 Web 端的能力。但项目中部分依赖在 Web 环境存在兼容性限制，需要根据实际使用场景选择合适的方案。

本文档分析三种可行的网页端方案，帮助做出技术决策。

---

## 二、Flutter Web 兼容性分析

| 依赖 | 移动端/桌面端 | Web 端 | 说明 |
|------|-------------|--------|------|
| sqflite（SQLite） | ✅ 原生支持 | ❌ 不支持 | Web 无原生 SQLite，需替换方案 |
| path_provider | ✅ 支持 | ⚠️ 有限 | Web 无文件系统概念，路径行为不同 |
| flutter_local_notifications | ✅ 支持 | ⚠️ 有限 | 需改用 Web Notification API |
| flutter_quill | ✅ 支持 | ✅ 支持 | 富文本编辑器本身支持 Web |
| file_picker | ✅ 支持 | ✅ 支持 | 直接可用 |
| image_picker | ✅ 支持 | ✅ 支持 | 直接可用 |
| 微信/QQ 登录 | ✅ 支持 | ⚠️ 需适配 | Web 登录流程与移动端不同 |

**核心结论**：Flutter Web 可以运行，但本地数据库和文件存储需要替换方案。

---

## 三、三种网页端方案对比

### 方案 A：移动端优先，不支持网页端

**描述**：仅编译 Android APK 和 Windows exe，不处理 Web 端。

| 维度 | 评估 |
|------|------|
| 开发工作量 | 最小，无需额外适配 |
| 运行成本 | 零成本，无需服务器 |
| 数据存储 | 本地 SQLite，完全离线可用 |
| 适用场景 | 个人自用，手机+电脑 |

**优点**：
- 开发最简单，按现有方案直接打包即可
- 零服务器成本，所有数据存在本地
- 无网络也能完整使用

**缺点**：
- 无法通过浏览器访问
- 换设备需手动导出/导入数据迁移

---

### 方案 B：Flutter Web + 本地存储（轻量离线）

**描述**：编译 Flutter Web，数据库改用 `hive`（纯 Dart，支持 Web），数据存在浏览器 IndexedDB。

| 维度 | 评估 |
|------|------|
| 开发工作量 | 中等，需替换数据库层 |
| 运行成本 | 低，仅需静态托管（GitHub Pages/Vercel 免费） |
| 数据存储 | 浏览器 IndexedDB，刷新不丢失 |
| 云端同步 | 可选，登录后才需要后端 |

**需要改动的代码**：

```yaml
# pubspec.yaml 调整
dependencies:
  # 移除或条件导入
  # sqflite: ^2.3.0
  
  # 新增 Web 兼容数据库
  hive: ^2.2.0
  hive_flutter: ^1.1.0
  
  # Web 端 SQLite（如仍需 SQL）
  sqflite_common_ffi_web: ^0.1.0
```

**优点**：
- 打开网页就能用，无需安装
- 刷新页面数据不丢失（IndexedDB 持久化）
- 可以配合 PWA 添加到桌面，体验接近原生
- 无网络时也能使用（基于本地存储）

**缺点**：
- 数据库层需要抽象，区分 Web 和移动端实现
- 图片存储在浏览器，大图片可能占用较多内存
- 换设备/换浏览器数据不互通

---

### 方案 C：纯在线 Web（无离线能力）

**描述**：网页端完全依赖后端 API，所有数据存在服务端，浏览器只做渲染。

| 维度 | 评估 |
|------|------|
| 开发工作量 | 较大，需完整后端 + API |
| 运行成本 | 需长期运行后端服务器（约 50-200 元/月） |
| 数据存储 | 服务端数据库（PostgreSQL/MySQL） |
| 网络要求 | 必须联网，否则无法使用 |

**架构变化**：

```
移动端：本地 SQLite ←→ 空闲时同步 → 后端 API
                            ↑
网页端：  浏览器 ←→ 直接读写 → 后端 API
                            ↓
                    服务端数据库（PostgreSQL）
```

**优点**：
- 数据实时一致，多端同时打开无冲突
- 换设备/换浏览器数据自动同步
- 网页端代码更简单（只需调 API，无本地存储逻辑）

**缺点**：
- 必须有 7×24 小时运行的后端服务
- 网页端无网即不可用
- 移动端和网页端两套数据逻辑，代码复用率低
- 个人自用维护成本高

---

## 四、方案详细对比表

| 对比项 | 方案 A<br>（仅移动端） | 方案 B<br>（Web + 本地存储） | 方案 C<br>（纯在线 Web） |
|--------|---------------------|---------------------------|------------------------|
| **开发难度** | ⭐ 低 | ⭐⭐ 中 | ⭐⭐⭐ 高 |
| **服务器成本** | 0 元 | 0 元（静态托管免费） | 50-200 元/月 |
| **无网络使用** | ✅ 完全可用 | ✅ 可用 | ❌ 不可用 |
| **多设备同步** | ❌ 手动导出导入 | ❌ 不支持 | ✅ 自动同步 |
| **浏览器访问** | ❌ 不支持 | ✅ 支持 | ✅ 支持 |
| **数据安全性** | 依赖本地备份 | 依赖本地备份 | 服务端自动备份 |
| **长期维护** | 几乎为零 | 低 | 需持续维护后端 |
| **适用人群** | 个人自用 | 个人自用 + 偶尔网页使用 | 多设备用户/团队协作 |

## 五、各方案推荐场景

### 推荐方案 A（仅移动端）的情况

- 仅自己使用，设备固定（1-2 部手机 + 1 台电脑）
- 不想维护服务器
- 希望完全离线也能使用
- 不介意换设备时手动导出/导入数据

**实施路径**：
1. 按现有方案开发
2. `flutter build apk --release` 生成安卓安装包
3. `flutter build windows --release` 生成 Windows 程序
4. 完成

---

### 推荐方案 B（Web + 本地存储）的情况

- 希望偶尔在浏览器中使用（如公司电脑不方便安装软件）
- 不介意数据存在浏览器，换设备需重新同步
- 不想承担服务器成本
- 能接受轻度适配工作

**实施路径**：
1. 开发时抽象数据库层，定义统一接口
2. 移动端实现：SQLite（sqflite）
3. Web 端实现：Hive / IndexedDB
4. 图片存储 Web 端改为 Base64 / ObjectURL
5. `flutter build web --release` 部署到静态托管

---

### 推荐方案 C（纯在线 Web）的情况

- 需要多设备实时同步
- 有团队协作或数据共享需求
- 愿意承担服务器成本
- 有后端开发能力或预算

**实施路径**：
1. 搭建后端服务（Node.js/NestJS + PostgreSQL）
2. 设计 REST API（增删改查、用户认证、文件上传）
3. 移动端：本地 SQLite + 空闲同步到后端
4. 网页端：纯 API 调用，无本地数据库
5. 分别部署移动端和网页端

---

## 六、决策建议

### 如果是个人自用

**首选方案 A**。理由：
- 最简单，开发完直接打包 APK 和 exe 即可
- 零运营成本，不需要服务器
- 完全离线可用，不依赖任何外部服务
- 数据完全掌握在自己手中

数据迁移问题可通过导出/导入功能解决，频率不高（换手机或重装系统时）。

---

### 如果未来可能给别人用

**先方案 A，后扩展方案 B**。理由：
- 第一阶段：完成移动端核心功能，自己先用起来
- 第二阶段：需要网页端时，将数据库层抽象，增加 Web 端的 Hive 实现
- 云端同步作为可选功能，需要时才接入后端

这样可以避免前期过度设计，按需扩展。

---

### 如果坚持要网页端且不想花钱

**选择方案 B**。理由：
- 静态托管免费（GitHub Pages、Vercel、Netlify 均可）
- 数据存在浏览器 IndexedDB，刷新不丢
- 配合 PWA 可以添加到桌面，体验接近 App

---

## 七、结论

| 问题 | 答案 |
|------|------|
| Flutter 能编译网页端吗？ | ✅ 能 |
| 网页端能用所有功能吗？ | ⚠️ 需要适配数据库和文件存储 |
| 个人自用需要网页端吗？ | ❌ 不需要，APK + exe 就够了 |
| 免费的网页端方案有吗？ | ✅ 方案 B（静态托管 + 浏览器存储） |
| 网页端和移动端能共享代码吗？ | ⚠️ UI 层可共享，数据层需抽象适配 |

**最终建议**：

> 对于个人自用场景，优先完成移动端和桌面端（方案 A）。网页端作为后续扩展项，在核心功能稳定后再评估是否需要投入。避免前期为了支持网页端而引入不必要的复杂度（后端服务、服务器维护、API 设计等）。
